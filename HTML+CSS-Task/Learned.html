How the web works

Tue. LAB2 08.07.2025

Deep dive into web architecture, protocols, and data flow
Modern web architectures, APIs, databases and rendering techniques

الإنترنت vs الويب
- الإنترنت: البنية التحتية للشبكات (الطرق والكابلات)
- الويب: نظام من المستندات والمواقع المرتبطة (حركة المرور على الطرق)
- مخترع الويب: تيم بيرنرز لي عام 1989

طبقات الويب
1. Surface Web: المحتوى المفهرس بواسطة محركات البحث
2. Deep Web: قواعد البيانات والمناطق المحمية بكلمة مرور
3. Dark Web: الشبكات الخاصة التي تتطلب بروتوكولات خاصة (مثل Tor)

السيرفرات والعناوين
- السيرفر: جهاز مخصص لاستقبال الطلبات وإرسال الردود
- IP Address: عنوان افتراضي (مثل 192.168.1.1)
- MAC Address: عنوان فيزيائي فريد للجهاز
- DNS: نظام يحول أسماء النطاقات (google.com) إلى عناوين IP

تحليل URL
مثال: https://www.example.com:443/path?query=value#fragment
1. البروتوكول (https)
2. النطاق (www.example.com)
3. المنفذ (443)
4. المسار (/path)
5. الاستعلام (?query=value)
6. الجزء المحدد (#fragment)

بروتوكولات الويب
- HTTP/HTTPS: لنقل صفحات الويب
- TCP/IP: لتقسيم البيانات وإعادة تجميعها
- FTP/SFTP: لنقل الملفات
- SMTP: لإرسال البريد الإلكتروني
- WebSocket: للاتصالات ثنائية الاتجاه

محركات البحث vs المتصفحات
- محركات البحث (Google): تقوم بـ:
  1. الزحف (Crawling)
  2. الفهرسة (Indexing)
  3. التصنيف (Ranking)
- المتصفحات (Chrome, Firefox): تعرض المحتوى وتنفذ JavaScript

طلب HTTP واستجابة
- Methods: GET, POST, PUT, DELETE, PATCH
- Status Codes:
  • 2xx: نجاح
  • 3xx: تحويل
  • 4xx: خطأ عميل
  • 5xx: خطأ سيرفر
- Headers: معلومات إضافية عن الطلب/الاستجابة
- Cookies: تخزين بيانات على جهاز المستخدم

JSON وتدفق البيانات
- JSON: تنسيق خفيف لتبادل البيانات (key-value pairs)
- أنواع APIs:
  1. RESTful: يعتمد على موارد HTTP
  2. GraphQL: يسمح بطلب بيانات محددة
  3. gRPC: عالي الأداء باستخدام Protocol Buffers
  4. SOAP: بروتوكول قديم يعتمد على XML

قواعد البيانات
- SQL (MySQL): منظمة في جداول مع علاقات
- NoSQL (MongoDB): مرنة بدون مخطط ثابت
- ORM: طبقة تربط بين لغة البرمجة وقاعدة البيانات

Frontend vs Backend
- Frontend: واجهة المستخدم (HTML, CSS, JS)
- Backend: معالجة البيانات (Node.js, Python, PHP)
- يتواصلان عبر طلبات HTTP وAPIs

أنواع واجهات برمجة التطبيقات (APIs)

1. RESTful API:
- تعتمد على موارد HTTP (GET/POST/PUT/DELETE)
- عديمة الحالة (Stateless)
- تبادل البيانات عادةً بصيغة JSON
- مثال: https://api.example.com/users

2. GraphQL:
- تسمح بطلب بيانات محددة فقط
- نقطة نهاية واحدة للكل الاستعلامات
- تقلل من كمية البيانات المنقولة
- مثال: query { user(id: 1) { name, email } }

3. SOAP:
- بروتوكول قديم يعتمد على XML
- يحتاج لوصف WSDL
- آمن ولكن ثقيل

4. gRPC:
- يستخدم Protocol Buffers بدلاً من JSON/XML
- سريع جداً ومناسب للخدمات الدقيقة (Microservices)
- يدعم الاتصال ثنائي الاتجاه

أنواع قواعد البيانات

1. العلائقية (SQL):
- MySQL, PostgreSQL
- البيانات منظمة في جداول مترابطة
- تستخدم لغة SQL للاستعلامات

2. NoSQL:
- MongoDB, Firebase
- مرنة بدون مخطط ثابت
- مناسبة للبيانات غير المنظمة

3. قواعد بيانات كائنية:
- تخزن البيانات ككائنات
- مناسبة للغات OOP مثل Java

4. قواعد بيانات بيانية:
- Neo4j
- تمثل البيانات كعقد وعلاقات
- مثالية للبيانات شديدة الترابط

تقنيات عرض الويب (Rendering)

1. Client-Side Rendering (CSR):
- يتم التحميل في المتصفح باستخدام JavaScript
- مثال: React, Vue, Angular
- إيجابيات: تجربة مستخدم سلسة
- سلبيات: تحسين محركات البحث (SEO) ضعيف

2. Server-Side Rendering (SSR):
- يتم التحميل على الخادم قبل الإرسال
- مثال: Next.js, Nuxt.js
- إيجابيات: SEO أفضل، تحميل أسرع للصفحة الأولى
- سلبيات: حمل أكبر على الخادم

3. Static Site Generation (SSG):
- يتم بناء الصفحات مسبقاً أثناء التطوير
- مثال: Gatsby, Jekyll
- إيجابيات: أداء ممتاز، أمان عالي
- سلبيات: غير مناسب للمحتوى الديناميكي

4. Incremental Static Regeneration (ISR):
- يجمع بين SSG و SSR
- يتم إعادة بناء الصفحات بشكل دوري
- مثال: Next.js مع ISR
- إيجابيات: أداء عالي مع محتوى محدث

أنواع تطبيقات الويب

1. Single Page Applications (SPAs):
- تحميل صفحة واحدة فقط مع تحديثات ديناميكية
- مثال: Gmail, Facebook
- تستخدم عادةً مع CSR

2. Multi-Page Applications (MPAs):
- تحميل صفحة جديدة لكل طلب
- مثال: المواقع التقليدية
- تستخدم عادةً مع SSR

3. The Vanillas (الطريقة التقليدية):
- HTML خام مع CSS و JavaScript بسيط
- أصبحت نادرة الاستخدام في التطبيقات الحديثة

التطور في واجهات الويب

الجيل القديم:
- jQuery وطرق تقليدية
- صفحات كاملة يتم إعادة تحميلها

الجيل الجديد:
- أطر عمل مثل React, Vue, Angular
- SSR الحديث (Next.js, Nuxt.js)
- تطبيقات متقدمة مع إدارة حالة معقدة

Data Types & CRUD Operations

Wed. LAB3 10.07.2025

Understanding JavaScript Data Types and Basic CRUD Operations after 2 hours of JS problem solving. We practiced object traversal, array-to-object transformation, value swapping, and grouped by properties using Map.

Read More... أنواع البيانات الأساسية في JavaScript

1. الأنواع البدائية (Primitive):
- Number: الأرقام (مثل 42 أو 3.14)
- String: النصوص (مثل "Hello")
- Boolean: القيم المنطقية (true/false)
- Undefined: عندما يكون المتغير غير معرّف
- Null: قيمة متعمدة لعدم وجود قيمة
- Symbol: قيمة فريدة وغير قابلة للتغيير (من ES6)

2. الأنواع غير البدائية (Non-Primitive):
- Object: لتخزين مجموعات من البيانات
- Array: لقوائم البيانات المرتبة
- Function: لتخزين الأكواد القابلة للتنفيذ

عمليات CRUD الأساسية

Create (إنشاء): let arr = [1, 2, 3];
Read (قراءة): console.log(arr[0]); // 1
Update (تحديث): arr[1] = 4; // [1, 4, 3]
Delete (حذف): delete arr[2]; // [1, 4, empty]

المصفوفات (Arrays) والعمليات عليها

إنشاء مصفوفة:
let fruits = ['تفاح', 'موز', 'برتقال'];

العمليات الأساسية:
- fruits.push('فراولة'); // إضافة في النهاية
- fruits.pop(); // إزالة آخر عنصر
- fruits.unshift('مانجو'); // إضافة في البداية
- fruits.shift(); // إزالة أول عنصر

طرق متقدمة:
- fruits.slice(1, 3); // استخراج جزء
- fruits.splice(1, 1, 'كيوي'); // إزالة وإضافة
- fruits.concat(['أناناس']); // دمج مصفوفات
- fruits.indexOf('موز'); // البحث عن عنصر
- fruits.sort(); // الترتيب
- fruits.reverse(); // عكس الترتيب
- fruits.forEach(item => console.log(item)); // التكرار

النصوص (Strings) والعمليات عليها

إنشاء نص:
let message = "مرحباً بالعالم";

العمليات الأساسية:
- message.length; // طول النص
- message[4]; // الوصول لحرف
- message.includes('مرحباً'); // البحث
- message.startsWith('م'); // التحقق من البداية
- message.endsWith('م'); // التحقق من النهاية

طرق متقدمة:
- message.split(' '); // تقسيم النص إلى مصفوفة
- message.substring(0, 6); // استخراج جزء
- message.replace('العالم', 'الجميع'); // استبدال
- message.toUpperCase(); // تحويل لأحرف كبيرة
- message.trim(); // إزالة المسافات الزائدة في النهاية والبداية
- `${message}!`; // قوالب النصوص (Template Literals)

تمارين تطبيقية

1. أنشئ مصفوفة أسماء وأضف اسم جديد ثم احذف الأول
2. ابحث عن كلمة معينة في نص واستبدلها
3. حول مصفوفة أسماء إلى نص مفصول بفواصل
4. استخرج الأرقام فقط من مصفوفة مختلطة

ما هو HTML؟
- لغة ترميز (Markup Language) وليست لغة برمجة
- وظيفتها هيكلة المحتوى على الويب
- تتكون من عناصر (Tags) مثل <h1> و <p>
- كلمة HyperText تشير إلى النصوص التفاعلية مثل الروابط
- تشكل مع CSS حوالي 60% من برمجة الواجهات الأمامية

نموذج DOM (Document Object Model)
- واجهة برمجة (API) تربط بين HTML وJavaScript
- تمثيل شجري (Tree Structure) لهيكل الصفحة
- يمكن تعديله ديناميكياً باستخدام JavaScript

العلامات الدلالية (Semantic Tags)
- مثل <header>، <footer>، <section>
- لا تختلف عن <div> في التنسيق ولكنها:
  • أكثر وضوحاً في قراءة الكود
  • أفضل لمحركات البحث (SEO)
  • تساعد في إمكانية الوصول (Accessibility)

نماذج الإدخال (Form Tags)
- مثل <form>، <input>، <select>
- تختلف عن Semantic Tags لأن لها تأثير وظيفي
- يمكن ربطها بمعالج أحداث مثل onSubmit

تطور HTML
- قبل HTML5: كان كل متصفح يعالج الكود بشكل مختلف
- بعد HTML5: أصبحت المواصفات موحدة
- أفضل الممارسات:
  • وضع سكربتات JavaScript في نهاية <body>
  • استخدام async للسيناريوهات الخاصة

نموذج الصندوق (Box Model)
- يمثل هيكل العنصر ومكوناته:
  1. المحتوى (Content)
  2. الحشوة (Padding)
  3. الحدود (Border)
  4. الهامش (Margin)
- وحدة rem: مرتبطة بحجم الخط في العنصر الأم

الفئات الزائفة (Pseudo-classes)
- تبدأ بعلامة : أو ::
- أمثلة:
  • :hover - عند تمرير المؤشر
  • :focus - عند التركيز على العنصر
  • ::before - لإضافة محتوى قبل العنصر
- استخدام *: للتعميم على كل العناصر

HTML في التطوير الحديث
- مكتبات مثل htmx:
  • تتيح تفاعلات معقدة بدون JavaScript كثيف
  • مقاربة لمفهوم React ولكن أخف وزناً
  • تحظى بشعبية متزايدة في 2025
- أطر عمل مثل React تستخدم JSX الذي يمزج HTML مع JavaScript

أفضل الممارسات

1. استخدام العلامات الدلالية المناسبة
2. فصل الهيكل (HTML) عن التنسيق (CSS) عن السلوك (JS)
3. التحقق من صحة الكود باستخدام أدوات مثل W3C Validator
4. كتابة كود قابل للقراءة مع تعليقات توضيحية
5. مراعاة إمكانية الوصول (Accessibility) منذ البداية

ReactJS & Hooks

Sun. LAB7 20.07.2025

Deep dive into React components lifecycle and essential hooks patterns

Read More... اليوم السابع: ReactJS - دورة الحياة والهوكس الأساسية

دورة حياة المكونات (Component Lifecycle)
المراحل الأساسية:
1. مرحلة ما قبل التصيير (Pre-render):
  - تحضير البيانات والمتغيرات
  - تنفيذ الأكواد خارج useEffect و useLayoutEffect

2. مرحلة التصيير (Rendering):
  - بناء الـVirtual DOM
  - تنفيذ return() الرئيسي

3. مرحلة الرسم (Painting):
  - تحديث الـDOM الحقيقي
  - عرض التغييرات للمستخدم

دور useEffect:
- يعمل بين مرحلة التصيير والرسم
- استخداماته الرئيسية:
  • جلب البيانات (Data Fetching)
  • الاشتراكات (Subscriptions)
  • التعديلات على DOM

أفضل الممارسات:
- تقليل dependencies لتجنب إعادة التصيير المتكرر
- استخدام cleanup function لإلغاء الاشتراكات:
useEffect(() => {
  const subscription = API.subscribe();
  return () => subscription.unsubscribe();
}, []);

إدارة الحالة مع useState
النمط الأمثل:
- استخدام دالة التحديث لتجنب المشاكل:
setCount(prevCount => prevCount + 1);

لماذا الدالة أفضل؟
1. تضمن الحصول على أحدث قيمة للحالة
2. تمنع مشاكل التزامن (Race Conditions)
3. تقلل من إعادة التصيير غير الضرورية

حالات خاصة:
- القيمة الأولية المعقدة:
const [data, setData] = useState(() => calculateInitialValue());

مبادئ SOLID في React
1. Single Responsibility:
- كل مكون يجب أن يكون له مسؤولية واحدة
- مثال: فصل مكون العرض عن مكون جلب البيانات

2. Open/Closed:
- المكون مفتوح للتوسعة مغلق للتعديل
- تحقيقها عبر: Composition, HOCs, Custom Hooks

3. Liskov Substitution:
- إمكانية استبدال المكونات ببدائلها
- مثال: مكون Button بأشكال مختلفة لكن نفس الوظيفة

4. Interface Segregation:
- تقسيم الـprops إلى واجهات متخصصة
- تجنب المكونات "السويسرية" ذات الـprops الكثيرة

5. Dependency Inversion:
- الاعتماد على التجريدات لا التنفيذ
- مثال: استخدام Context API بدلاً من props drilling

useContext للإدارة المركزية
خطوات التنفيذ:
1. إنشاء Context:
const ThemeContext = createContext('light');

2. استخدام Provider:
<ThemeContext.Provider value="dark">
  <App />
</ThemeContext.Provider>

3. الاستهلاك:
const theme = useContext(ThemeContext);

محظورات الاستخدام:
- داخل شروط (if/else)
- داخل حلقات (loops)
- خارج مكونات React
- داخل JSX مباشرة

أفضل الممارسات:
- تجنب إساءة الاستخدام للإدارة العالمية المعقدة
- استخدام مع Reducer للعمليات المعقدة
- تقسيم الـContexts حسب المجالات (Theme, Auth, etc.)

أمثلة وتطبيقات عملية

نظام الثيمات:
const ThemeContext = createContext();

function App() {
  const [theme, setTheme] = useState('light');
  return (
   <ThemeContext.Provider value={{ theme, setTheme }}>
    <Header />
    <Content />
   </ThemeContext.Provider>
  );
}

إدارة حالة النموذج:
const [form, setForm] = useState({ email: '', password: '' });

const handleChange = (e) => {
  setForm(prev => ({ ...prev, [e.target.name]: e.target.value }));
}

2. نفذ نموذج تسجيل مع إدارة الحالة المتقدمة
3. قسم مكون معقد إلى مكونات صغيرة تطبق SOLID
4. أنشئ custom hook لإدارة API calls
5. حول مشروع من useState فقط إلى استخدام useContext

المكونات (Components) وJSX
- JSX يمزج HTML مع JavaScript
- يساعد محررات الأكواد باقتراحات HTML
- يجب أن تبدأ أسماء المكونات بحرف كبير
- مثال:
function MyComponent() {
  return <div>Hello World</div>;
}

دورة حياة المكونات (Component Lifecycle)
1. مرحلة ما قبل التصيير (Pre-render)
2. مرحلة التصيير (Rendering)
3. مرحلة الرسم (Painting)
- تعمل useEffect بين المرحلتين الثانية والثالثة
- تعمل useLayoutEffect قبل مرحلة التصيير
- كثرة dependencies قد تسبب إعادة تصيير متكررة

قواعد استخدام الهوكس (Hooks)
1. لا تستخدم خارج مكونات React
2. لا تستخدم داخل شروط أو loops
3. المسموح خارج المكونات:
  - import/export
  - المتغيرات العامة (Global Variables)
4. useRef الوحيد الذي يعمل خارج دورة التصيير
5. محظور استخدام Hooks في: الشروط، خارج المكونات، أو داخل JSX مباشرة

الهوكس الأساسية
useState:
- لإدارة الحالة المحلية
- يحدث القيم ويعيد التصيير بدون refresh
- الأفضل استخدام دالة داخل setState:
setCount(prev => prev + 1);
- مثال: const [count, setCount] = useState(0);

useEffect:
- لدورة حياة المكون (Mount-Update-Unmount)
- استخدامات رئيسية:
  • جلب البيانات (Data Fetching)
  • الاشتراكات (Subscriptions)
  • التعديلات اليدوية على DOM
- مثال:
useEffect(() => {
  fetch('/api/data')
  .then(response => response.json())
  .then(data => setData(data));
  return () => { /* Cleanup */ };
}, []);

useLayoutEffect:
- يعمل قبل التصيير مباشرة
- مفيد للقياسات والتعديلات على DOM

useActionState (React 19 الجديدة):
- مصممة خصيصاً للنماذج (Forms)
- تدمج إدارة الحالة مع معالجة الأحداث
- مثال:
const [state, action, pending] = useActionState(async (prevState, formData) => {
  // Handle form submission
}, initialState);

useContext للإدارة المركزية
- لمشاركة البيانات بين المكونات بدون تمرير props
- خطوات الاستخدام:
1. إنشاء Context:
const MyContext = createContext();
2. استخدام Provider:
<MyContext.Provider value={data}>...</MyContext.Provider>
3. الاستهلاك في المكونات:
const data = useContext(MyContext);
- مثال عملي: أنظمة Themes، إدارة المستخدم

- API وهمي مجاني للتجارب
- مثال: fetch('https://jsonplaceholder.typicode.com/posts')

Axios:
- مكتبة أكثر قوة من fetch
- مزايا:
  • تحويل تلقائي لـ JSON
  • معالجة أخطاء أفضل
  • إلغاء الطلبات
  • تتبع تقدم الطلبات
- مثال:
axios.get('https://api.example.com/data')
  .then(response => console.log(response.data))
  .catch(error => console.error(error));

مبدأ SOLID في React
1. Single Responsibility: كل مكون له وظيفة واحدة
2. Open/Closed: قابل للتوسعة دون تعديل
3. Liskov Substitution: قابل للاستبدال
4. Interface Segregation: تقسيم الواجهات
5. Dependency Inversion: الاعتماد على التجريدات
- تطبيق هذه المبادئ يحسن من جودة وسهولة صيانة الكود
